# Оплата события организатором и подтверждение оплаты площадкой

Документ описывает проектируемый флоу оплаты бронирования события со стороны организатора и подтверждение оплаты администратором площадки.

## Цели
- Ввести статусы «оплачено пользователем» и «подтверждено админом площадки» для бронирований событий.
- Обеспечить выбор метода оплаты на странице события с контентом по типу метода.
- Добавить подтверждение оплаты с доказательством (скриншот и/или комментарий).
- Зафиксировать механизмы хранения и аудита.

## Роли и права
- Организатор события (создатель/админ события): может выбрать метод оплаты и отправить запрос «Оплачено».
- Администратор площадки: любой пользователь с активным контрактом и правом подтверждения бронирования + admin.
- Доступ к доказательствам оплаты: пользователи с активным контрактом и правом работы с бронированиями.

## Источники данных оплаты
- Используются методы оплаты, закрепленные за контрактом (владелец/супервайзер), выбранные в настройках площадки.
- Для бронирования фиксируются снимки получателя и методов оплаты (уже реализовано в бронированиях).
- На странице события отображаются методы оплаты из снимка бронирования.

## Статусы
### Статусы подтверждения оплаты бронирования
- `none` — подтверждение оплаты не запрошено.
- `user_paid_pending` — «оплачено пользователем», ожидает подтверждения администратором.
- `user_paid_rejected` — администратор отклонил оплату.
- `admin_confirmed` — «оплата подтверждена админом площадки».

### Правила переходов
- Организатор может перевести `none` → `user_paid_pending`, указав скриншот и/или комментарий.
- Администратор может перевести `user_paid_pending` → `admin_confirmed` или `user_paid_rejected`.
- После `user_paid_rejected` организатор может повторно отправить запрос `user_paid_pending`.
- Если у площадки включено «мгновенное подтверждение», то при `admin_confirmed` бронирование переводится в `approved`.

## UI‑флоу (страница события)
1. Блок «Оплата бронирования» доступен только после перевода заявки в статус «Ожидает оплату» (действие админа площадки).
2. Блок показывает список доступных методов оплаты из снимка бронирования.
3. Организатор выбирает метод оплаты:
   - СБП: показываются телефон/карта и отображаемое имя получателя, инструкция и сумма к оплате.
   - Эквайринг/Баланс: пока заглушки (показывается уведомление «способ не поддерживается»).
4. Кнопка «Оплатить» доступна только при выборе поддерживаемого метода.
5. При нажатии «Оплатить» открывается confirm‑попап:
   - прикрепление скриншота оплаты (файл);
   - комментарий (textarea);
   - требуется заполнить хотя бы одно из двух.
6. После отправки показывается статус «Ожидает подтверждения».
7. Администратор площадки подтверждает/отклоняет оплату на странице бронирований площадки.

## Механизм хранения
### Таблица подтверждений оплаты
`booking_payment_confirmations`
- `id`
- `event_booking_id` (или `booking_id` — точное имя зависит от модели)
- `payment_method_id` (выбранный метод)
- `payment_method_snapshot` (JSON: тип, название, реквизиты)
- `evidence_comment` (nullable)
- `evidence_media_id` (nullable, ссылка на Media)
- `status` (`pending`, `approved`, `rejected`)
- `requested_by_user_id`
- `decided_by_user_id` (nullable)
- `decided_at` (nullable)
- `decision_comment` (nullable)
- `created_at`, `updated_at`

### Поля в бронировании
- `payment_confirm_status` (`none`, `user_paid_pending`, `user_paid_rejected`, `admin_confirmed`)
- `payment_confirmed_at` (nullable)
- `payment_last_confirmation_id` (nullable, FK)

> Примечание: Статус бронирования — агрегированный, полная история находится в `booking_payment_confirmations`.

## Флоу админа площадки (страница бронирований)
1. Новая заявка в статусе `pending` открывается через «Просмотр заявки».
2. В попапе админ задаёт параметры оплаты (порядок оплаты, срок ожидания, стоимость, комментарий) и нажимает «В оплату».
3. После этого заявка переводится в `awaiting_payment`, организатор получает доступ к оплате.
4. Для заявок в оплате «Просмотр заявки» открывает информационный попап без редактирования.
5. Кнопки «Оплачено» нет — подтверждение оплаты делается через блок «Подтверждение оплаты».
6. Изменённая стоимость в попапе применяется к сумме оплаты и сохраняется в `Payment`.

## Уведомления
- При создании запроса «Оплачено» — уведомление администраторам площадки.
- При подтверждении/отклонении — уведомление организатору события.

## Ограничения по времени
- Возможность «Оплачено» зависит от настроек площадки.
- Предлагается добавить отдельную настройку в `VenueSettings`, например:
  - `payment_confirmation_before_start_minutes` (0 = без ограничений).

## Ошибки и граничные случаи
- Метод оплаты деактивирован: нельзя выбрать для нового запроса, но история подтверждений сохраняется.
- Отмененное/перенесенное бронирование: подтверждения остаются, возвраты — в будущем статусе `refunded`.
- Эквайринг/баланс: пока только уведомление, без создания запросов.

## Аудит
- Все изменения подтверждений и статусов фиксируются аудитом.
- Доступ к доказательствам по правам работы с бронированиями.

## Реализация (backend)
- Таблица `booking_payment_confirmations` и поля в `event_bookings` для агрегированного статуса.
- Сервис подтверждения оплаты с валидацией метода и доказательств (скриншот/комментарий).
- Эндпоинты для отправки запроса, подтверждения и отклонения оплаты.
- Уведомления по запросу и решению.
