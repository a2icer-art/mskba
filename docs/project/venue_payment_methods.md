# Методы оплаты бронирований площадок

## Связанные материалы
- Задача: `docs/tasks/114/task.md`
- Коммиты: `ffeb5c69` (реализация), `c0b3b3f9` (проектирование)

## Термины
- **Получатель оплаты** — сторона, чьи реквизиты показываются пользователям при оплате бронирования.
- **Источник реквизитов** — откуда берутся реквизиты (супервайзер, владелец, площадка).
- **Метод оплаты** — способ оплаты (СБП, баланс MSKBA, эквайринг).

## Правила выбора получателя и реквизитов
1. В настройках площадки задаётся `payment_recipient_source` (auto/supervisor/owner/venue).
2. Если выбранный источник недоступен (нет активного контракта или нет активных методов оплаты), используется автоматический режим.
3. В режиме auto приоритет такой:
   - активный контракт супервайзера + активные методы оплаты контракта;
   - активный контракт владельца + активные методы оплаты контракта;
   - методы оплаты площадки.
4. Если активных методов оплаты нет ни у одного источника — бронирования с предоплатой недоступны, пользователю показывается сообщение «Оплата не настроена».

### Ответы на кейсы
- **Есть супервайзер, нет владельца:** получатель — супервайзер, показываются методы оплаты контракта супервайзера.
- **Нет супервайзера, есть владелец:** получатель — владелец, показываются методы оплаты контракта владельца.
- **Нет владельца и супервайзера:** используется профиль оплаты площадки (фолбэк обязателен).

## Должны ли быть методы оплаты у площадки?
Ниже — ключевые сценарии и вывод.

### Сценарий A: площадка без владельца и супервайзера
Если полагаться только на владельца/супервайзера, то предоплата становится невозможной, а процесс бронирования блокируется. В реальной практике такой статус может сохраняться длительное время (новая площадка, конфликт по контрактам, задержка модерации).

### Сценарий B: супервайзер есть, владелец отсутствует
Супервайзер может принимать оплату, но при его увольнении/замене процесс снова остановится. Площадка в этот момент остаётся без «дефолтных» реквизитов.

### Сценарий C: владелец есть, но не активен/не отвечает
Даже при наличии владельца, его методы оплаты могут быть выключены или не настроены. Без резервного варианта бронирования становятся нестабильными.

### Сценарий D: переходы между источниками
При смене контракта (владелец/супервайзер) важно не терять возможность оплаты между событиями. Площадка как фолбэк снижает риски простоев.

### Вывод
Рекомендуется хранить методы оплаты на уровне площадки как обязательный фолбэк. Это снижает риски блокировки предоплат и не зависит от кадровых изменений. Даже если в большинстве случаев есть владелец или супервайзер, площадка остаётся стабильной точкой ответственности.

## Модель данных (проект)
### payment_methods
Полиморфная таблица методов оплаты, привязанных к площадке или контракту.

Поля:
- `id`
- `owner_type` (`Venue` / `Contract`)
- `owner_id`
- `type` (enum: `sbp`, `balance`, `acquiring`)
- `label` (отображаемое название)
- `phone` (только для СБП)
- `display_name` (только для СБП, имя/инициалы для проверки)
- `is_active`
- `sort_order`
- `meta` (json, для будущих расширений)
- `created_by`, `updated_by`

### venue_settings
Добавить:
- `payment_recipient_source` (enum: `auto`, `supervisor`, `owner`, `venue`) — предпочтительный источник реквизитов.

### booking/payment snapshot
При формировании бронирования фиксировать:
- `payment_recipient_type`/`payment_recipient_id` (Contract/Venue)
- `payment_recipient_label`
- `payment_methods_snapshot` (список активных методов на момент бронирования)

## Правила доступности методов
- **СБП:** активен только при заполненных `phone` и `display_name`.
- **Баланс MSKBA:** активен, если включён на площадке и у пользователя достаточно средств; иначе показывать как недоступный с причиной.
- **Эквайринг:** активен как флаг, но в UI обозначается как «скоро» (заглушка).
- **Удаление:** запрещено, если метод используется в текущих бронированиях (pending/awaiting_payment/paid/approved).

## Интерфейс управления
### Настройки площадки
Раздел «Оплата» в `/venues/<type>/<venue>/admin/settings`:
- выбор источника реквизитов (auto/supervisor/owner/venue) с подсказками по фолбэку;
- список методов оплаты площадки (добавить/редактировать/выключить/упорядочить).

### Супервайзер
Модальное окно управления методами оплаты контракта супервайзера (привязка к контракту):
- доступ: пользователь контракта, пользователь с правом редактирования данного контракта, admin;
- методы оплаты не смешиваются с площадкой.
- точка входа: список контрактов площадки (действие в строке контракта).
- редактирование выполняется inline внутри карточки метода; форма добавления — отдельный блок ниже списка.

### Владелец
Модальное окно управления методами оплаты контракта владельца (привязка к контракту):
- доступ: пользователь контракта, пользователь с правом редактирования данного контракта, admin;
- методы оплаты не смешиваются с площадкой.
- точка входа: список контрактов площадки (действие в строке контракта).
- редактирование выполняется inline внутри карточки метода; форма добавления — отдельный блок ниже списка.

## Отображение в бронировании
- В карточке бронирования показывать «Получатель оплаты» и его методы.
- Отображать только активные методы, с деталями (СБП: телефон, имя).
- Если оплата недоступна — показывать понятный статус и причину.
- Сообщение для пользователя: «Оплата не настроена. Обратитесь к ответственному: {контакт}, иначе — в техподдержку сайта».

## Примечания
- Профиль оплаты площадки обязателен как фолбэк, иначе предоплата блокируется.
- Комиссия супервайзера остаётся частью стоимости бронирования и не влияет на выбор получателя.
