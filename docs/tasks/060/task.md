# task_status
[100%]

# task_init_description
Применить настройки площадки (lead time, min interval) в логике бронирований вместо хардкода.

# task_short_description
Бронирования: использовать настройки площадки для lead time и минимального интервала.

# task_full_description
Нужно заменить захардкоженные проверки времени в бронированиях (lead time и минимальный интервал) на значения из настроек площадки (venue_settings). Проверки используются при создании бронирования и при проверках на стороне UI (скрытие кнопки). Требуется аккуратно использовать дефолтные значения, если настроек ещё нет.

# План выполнения задачи
1. Подготовить доступ к настройкам площадки при бронировании (получение settings/дефолтов). (subtasks/01_settings_access.md)
2. Обновить сервис бронирования и контроллеры, чтобы использовать lead time и min interval из настроек. (subtasks/02_booking_rules.md)
3. Обновить UI-логику скрытия кнопки бронирования на странице события, учитывая lead time. (subtasks/03_ui.md)
4. Обновить документацию задачи и статус, добавить шаги проверки. (subtasks/04_docs.md)

Алгоритм проверки в UI:
1. Открыть площадку и перейти в "Настройки" → задать lead time и min interval, сохранить.
2. На странице события открыть бронирование и проверить серверные ошибки:
   - если до начала события меньше lead time → получить ошибку о дедлайне;
   - если длительность меньше min interval → получить ошибку о длительности.
3. В попапе создания события с выбранной площадкой проверить, что lead time учитывается.
4. Создать бронирование в допустимых пределах — сохранение проходит успешно.
