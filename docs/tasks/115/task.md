# full_description
Нужно спроектировать детальный бизнес‑процесс оплаты события со стороны организатора: какие статусы вводим, кто и в какие моменты их меняет, как это отображается на странице события и как подтверждается оплата со стороны площадки. Обязательно предусмотреть выбор метода оплаты на странице события с разным содержимым по типу (например, для СБП показывать номер телефона/карты и отображаемое имя получателя). Должна быть кнопка «Оплачено», которая открывает confirm‑попап с возможностью прикрепления скриншота оплаты и текстовым комментарием, при этом одно из двух обязательно. Также нужно продумать механизм хранения этих данных в БД (статусы, подтверждение, файлы/комментарии, история). В результате требуется сформировать единый флоу и список сценариев, которые будут дальше реализовываться.
# План выполнения задачи
1. Собрать текущие сущности и процессы: события, бронирования, оплаты, методы оплаты, роли по контрактам — описать точки интеграции в новом флоу. (см. `docs/tasks/115/subtasks/01-context.md`)
2. Описать статусы и переходы: «оплачено пользователем» и «подтверждено админом площадки», кто может переводить, условия и ограничения. (см. `docs/tasks/115/subtasks/02-statuses.md`)
3. Проработать UI‑флоу на странице события: выбор метода оплаты, отображение данных по типам (СБП и т.д.), кнопка «Оплачено», confirm‑попап со скриншотом/комментарием (обязательность одного из двух). (см. `docs/tasks/115/subtasks/03-ui.md`)
4. Спроектировать хранение в БД: структура/поля для статусов, подтверждения, вложений, комментариев, истории изменений и аудит. (см. `docs/tasks/115/subtasks/04-storage.md`)
5. Сформировать список сценариев/кейсов (включая ошибки и спорные ситуации) и итоговые требования. (см. `docs/tasks/115/subtasks/05-scenarios.md`)

# Уточнения и решения
- Оплата по СБП: пользователь не может просто нажать «Оплачено» — обязателен скриншот или комментарий.
- Подтверждение касается бронирования.
- Организатор может повторно отправить запрос на подтверждение, если предыдущий отклонён администратором.
- Подтверждать могут: любой активный контракт с правом подтверждения бронирования + admin.
- Метод «баланс» пока заглушка: при выборе показывать уведомление «способ не поддерживается».
- При отмене/переносе: пока без возвратов; ожидаем будущий статус refunded со стороны администратора площадки.
- Медиа: хранить в общей системе медиа; доступ к просмотру — у пользователей с активным контрактом и правом работы с бронированиями.
- Уведомления о смене статусов обязательны.
- Ограничения по времени «оплачено» зависят от настроек площадки.
- Минимальных требований к контенту нет, так как решение подтверждает/отклоняет администратор, а организатор может отправить новый запрос.
